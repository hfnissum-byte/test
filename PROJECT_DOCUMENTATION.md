# QuantumProfit-Bot - End-to-End Project Documentation

This document describes the repository as it exists in `quantumprofit-bot/` today:
data ingestion, pattern-intelligence engine, embedding storage (vector DB),
quantum-inspired model, classical benchmark model, PPO (optional),
backtesting (deterministic baseline with pluggable policies),
and the always-on `continual-learning` loop.

Trading scope (current implementation):
- Markets: spot `BTC/USDT` (configured via `config.yaml`)
- Strategy direction: long/short signals are generated by policies, but the Phase-9 baseline
  backtest engine uses those signals as trade directions (no martingale/grid behavior).

Folders in this repo (high-level):
- `data/`: OHLCV download, incremental update, storage, live feed fallback
- `patterns/`: candlestick + chart/structure pattern detection, labeling, embeddings, vector DB memory
- `quantum_model/`: quantum-inspired hybrid estimator (with classical fallback when deps are missing)
- `classical_model/`: Transformer + LSTM sequence estimator (with sklearn fallback when deps are missing)
- `backtest/`: Phase-9 deterministic backtest engine + metrics + comparison
- `rl_agent/`: PPO training environment + PPO wrapper (optional, not used in baseline backtests)
- `continual/`: always-on incremental learning orchestration loop
- `utils/`: logging, time utilities, validation helpers
- `tests/`: unit tests for core pipeline components

## 2) Data Layer (data/)

Goal:
- Use real exchange OHLCV from CCXT
- Earliest-to-today historical download
- Incremental updates for always-on usage
- Store candles with dedup/upsert semantics

### 2.1 Downloader: `data/downloader.py`
Implements `CCXTDownloader`:
- Spot OHLCV via CCXT
- Primary exchange: `exchanges.primary.id` (default `binance`)
- Fallback exchange: `exchanges.fallback.id` (default `bybit`)
- Pagination:
  - Uses `since_ms` + timeframe duration
  - Detects stalled pagination and stops to avoid infinite loops
- Retries:
  - Uses `tenacity` when available, otherwise an internal exponential backoff loop

Main entrypoints:
- `download_earliest_to_today(symbols, timeframes)`
- `download_incremental(symbols, timeframes)`

### 2.2 Storage: `data/storage.py`
Implements `ParquetCandleStorage`:
- Stores candles by `(exchange, symbol, timeframe)` on disk
- Upsert/dedup:
  - Dedupes by `(exchange, symbol, timeframe, timestamp)` keeping the most recent record
- Backend:
  - Parquet when `pyarrow` is installed
  - CSV fallback when `pyarrow` is not installed (critical path remains runnable)

Key methods used across the pipeline:
- `upsert_candles(candles)`
- `get_latest_timestamp(exchange, symbol, timeframe)`
- `load_candles(exchange, symbol, timeframe, start_timestamp, end_timestamp)`

### 2.3 Live feed: `data/live_feed.py`
Implements a Binance WebSocket kline stream plus REST polling fallback:
- Intended to push closed candles only (partial candles skipped)
- Reconnects with configurable backoff

Important:
- The `continual-learning` loop uses the REST incremental update path for correctness.
- Live feed can be started via `update-data --live` (Phase 1 behavior).

## 3) Pattern Intelligence Engine (patterns/)

Goal:
- Detect candlestick + chart/structure patterns
- Extract geometry metrics (curve/shape aware)
- Label each detected instance with forward return outcomes (Good/Bad/Neutral where applicable)
- Produce embeddings and store them in a vector DB (Chroma/FAISS optional; local fallback guaranteed)

Entry point used by the CLI and continual loop:
- `patterns/intelligence_engine.py::build_pattern_db()`

### 3.1 Pattern detection
The engine runs detectors across:
- Candlestick patterns: (e.g. Hammer, Doji, Engulfing, Morning Star, etc.)
- Chart/structure patterns: double/triple tops, head-and-shoulders, wedges, triangles, flags, rectangles, channels, cup/handle, breakout+retest, etc.
- Geometry/curve awareness:
  - Pivot extraction + local extrema
  - Tolerance thresholds and symmetry/asymmetry measures
  - Curve window embedding context

Notes:
- Structural detectors use pivots + tolerance + confirmation logic; they are not reduced to trivial hardcoded toy rules.

### 3.2 Labeling
Each detected `PatternRecord` contains forward-return labels:
- next 1h return
- next 4h return
- next 24h return

Labels are derived using quantile-based outcome bucketing (configurable in the pipeline code).

### 3.3 Embeddings: `patterns/embeddings.py`
Embeddings are derived from:
- Pattern type
- Detector features
- Geometry metrics (shape context)
- Volatility context

Embedding dimension is fixed by the codebase (see `patterns/embeddings.py`).

### 3.4 Vector DB Memory: `patterns/vector_db.py`
`create_vector_db()` selects a backend:
- Chroma if installed and the collection is populated
- FAISS if installed
- Otherwise a local cosine-similarity memory is used

Important correctness detail:
- If Chroma is installed but the collection is empty, it is treated as unavailable so the pipeline falls back.

### 3.5 Output artifacts
For each symbol, the engine writes:
- `data/patterns/pattern_records_<SYMBOL>.csv`
  - includes pattern metadata, forward returns, embedding vector, confidence
- Vector DB artifacts in `data/patterns/vector_db/`

## 4) Model Stack

There are two baseline signal generators used for Phase-9 backtesting:
- Quantum-inspired estimator (quantum_model/)
- Classical benchmark sequence model (classical_model/)

PPO exists but is optional and NOT used as the Phase-9 baseline policy.

## 4.1 Quantum-Inspired Model (quantum_model/)

Goal:
- Use a hybrid quantum-classical estimator when PennyLane/PyTorch are available
- Fall back gracefully to a classical estimator when quantum deps are missing

Main files:
- `quantum_model/hybrid_model.py`: estimator implementation + fallback logic
- `quantum_model/trainer.py`: deterministic training orchestration
- `quantum_model/inference.py`: load + predict API

Training API used by CLI:
- `train_quantum_model(pattern_csv_path, out_dir, symbols, timeframes, horizon, max_rows, seed)`

Saved artifacts:
- `quantum_model.joblib`
- `quantum_model_meta.json` (auditable training metadata)

---

## 4.2 Classical Benchmark (classical_model/)

Goal:
- Transformer + LSTM hybrid estimator over sequences of embedding vectors
- Deterministic sklearn fallback when torch is unavailable

Main files:
- `classical_model/features.py`: builds sequence windows from pattern records
- `classical_model/transformer_lstm.py`: estimator implementation + fallback logic
- `classical_model/trainer.py`: deterministic orchestration
- `classical_model/inference.py`: load + predict API

Training API used by CLI:
- `train_classical_model(pattern_csv_path, out_dir, symbols, timeframes, horizon, seq_len, stride, ...)`

Saved artifacts:
- `classical_model.joblib`
- `classical_model_meta.json`

## 6) Backtesting (backtest/)

Goal:
- Deterministic baseline comparison across policies
- Pluggable policies (quantum, classical, pattern-similarity; PPO optional later)
- Realistic execution modeling:
  - fees
  - slippage
  - risk-based position sizing
  - ATR-based SL/TP with volatility-regime multipliers
  - single-position, non-overlapping trades (no overlap by design)
- Outputs comprehensive reports per policy:
  - `equity_curve.csv`
  - `drawdown_curve.csv`
  - `trades.csv`
  - `metrics.json`

Main files:
- `backtest/engine.py`: deterministic backtest + policy loop + report writer
- `backtest/policies.py`: `QuantumModelPolicy`, `ClassicalModelPolicy`, `PatternSimilarityPolicy`
- `backtest/metrics.py`: Sharpe, Sortino, Calmar, drawdown, win/loss, expectancy, etc.
- `backtest/compare.py`: ranks saved runs by Sharpe/net return

Core entrypoints:
- `backtest_and_compare(storage, pattern_csv_path, classical_model_path, quantum_model_path, vector_db_base_dir, cfg, policies=None)`
- `rank_backtest_policies(report_base_dir)`

Baseline policy set:
- quantum
- classical
- pattern-similarity

PPO:
- Exists as a policy option later, but is not default in baseline comparisons.

## 7) Continual Learning Loop (continual/)

Goal:
- Run an always-on loop that:
  1) updates stored OHLCV via REST incremental update
  2) rebuilds patterns + embeddings when a new closed candle is available for trigger timeframes
  3) retrains quantum + classical models deterministically
  4) optionally runs the Phase-9 baseline backtest again

Key file:
- `continual/runner.py::run_continual_learning()`

### 7.1 Learning trigger timeframes
By default:
- if both `1d` and `4h` exist in configured timeframes, the engine rebuilds on those closes only
Override:
- `--pattern-detection-timeframes` lets you specify trigger timeframes explicitly (e.g. `1h 15m`)

Cost warning:
- Triggering on lower timeframes (especially `5m`) greatly increases the number of pattern instances to evaluate.

### 7.2 What you should see in logs
When a cycle triggers, logs are expected in this order:
1) `continual ... building pattern DB ...`
2) `continual ... building patterns for <symbol>`
3) `... Wrote <N> pattern records ...` and `... Vector DB upserted ...`
4) `continual ... training quantum model for <symbol>`
5) `continual ... training classical model for <symbol>`
6) `continual ... running baseline backtest ...` (if not disabled)

If the loop says:
- `continual cycle=X no new candles; sleeping 300.0s`
then the trigger timeframes did not receive a new closed candle yet.

### 7.3 New CLI subcommand
Added command:
- `python main.py continual-learning ...`



```bash

```

- Baseline backtest requires stored candles for the configured execution timeframe (default `5m`).

Rank saved results:
```bash
python main.py --config config.yaml backtest-rank --base-dir backtest/results
```

### 8.6 Continual learning (always-on)
Run indefinitely:
```powershell
python main.py --config config.yaml continual-learning --max-cycles 0
```
Smoke test (one cycle, no backtest):
```powershell
python main.py --config config.yaml continual-learning --max-cycles 1 --no-backtest --poll-interval-seconds 5
```

Trigger rebuild on `1h/15m` closes:
```powershell
python main.py --config config.yaml continual-learning --max-cycles 0 --pattern-detection-timeframes 1h 15m
```

Disable backtest inside continual loop:
```powershell
python main.py --config config.yaml continual-learning --max-cycles 0 --no-backtest
```

## 9) Operational Notes & Troubleshooting

### 9.1 Why you might see `stalled incremental pagination`
The downloader includes a safety check:
- if the exchange returns the same last timestamp repeatedly (`last_ts <= current_since`)
then it stops that pagination loop to avoid an infinite loop.

In incremental mode, this is usually harmless because upserts still work for any candles returned.

### 9.2 Why continual cycle may skip rebuilding
The continual loop rebuilds only when a new closed candle is available for the configured trigger timeframes.
If your last update only produced new `5m` candles but the trigger set is `1d/4h`, then the loop will sleep until `1d/4h` closes are detected.

### 9.3 Chroma vector DB warnings
You may see warnings like:
- `Vector DB backend not available; falling back to local store. last_err=Chroma collection is empty`

This is a correctness feature:
- it prevents “empty Chroma” from causing pattern-similarity to return zero results.
- local cosine fallback remains functional.

### 9.4 Compute cost controls
Pattern detection on smaller timeframes can explode the number of detected instances.
Recommended starting learning triggers:
- default: `1d` and `4h`
- intermediate: `1h` (and optionally `15m` only if you accept higher compute)
- avoid: `5m` as the trigger until you have compute budget and validation thresholds
